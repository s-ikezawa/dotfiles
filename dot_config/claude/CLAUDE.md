# Web 調査の方針（全セッション適用）

WebFetch は取得したページを **小型の高速モデルで要約・抽出した結果**しか返さない（設計上ロッシー）。
そのため「ページ後半にある反証・結論」を取りこぼし、誤った内容を自信満々に報告する事故が起きうる。
これを避けるため、**Web 検索・調査を行うときは常に以下を守る**こと。重要かどうかで区別しない。

## 必須ルール

1. **全文取得を原則とする** — WebFetch の要約に頼らない。本文は決定論的な抽出器（trafilatura）で取得し、 モデルによる要約を挟まずに原文を直接読む。
2. **敵対的に検証する** — 「この主張は誤りである」という前提で反証を探す。複数ソースで突き合わせ、 食い違いは隠さず明示する。単一ソースの断定を鵜呑みにしない。
3. **一次情報まで遡る** — AI 生成ブログ・まとめ記事が氾濫しているため正確性が低い。 公式ドキュメント・公式リポジトリ・論文・一次データなど元ソースを必ず当たる。引用元を明示する。
4. **不確実なら不確実と言う** — 確認できていない点・利益相反のあるソース（作者の自己ベンチマーク等）は その旨を添える。確信度を偽らない。

## 取得パイプライン（検証済み 2026-06）

通常のページ（サーバーレンダリング / 記事・ブログ）:

```sh
# 本文だけを Markdown 抽出（JS・広告・ナビ等のボイラープレートを決定論的に除去）
curl -sL "<URL>" | trafilatura --markdown
```

- ツール: **trafilatura**（`trafilatura` で直接呼べる。導入済み）。独立系 ScrapingHub article-extraction-benchmark で本文抽出 F1 トップ層（Readability.js / newspaper4k / goose3 を上回る）。
- 注意: `-i` フラグは **URL リストファイル用**でローカル HTML ファイルではない。 ローカル HTML を処理するときは `cat file.html | trafilatura --markdown` のように **stdin** で渡す。

JS 描画（SPA）で trafilatura が空を返す場合は、無理に別手段（外部サービス等）を使わず、 **取得できなかった URL の一覧としてユーザーに報告する**。憶測で内容を埋めない。

## 重い調査

複数ソースを横断し主張を検証する本格調査は `/deep-research` スキルを使う （ファンアウト検索 → 全文取得 → 3票の敵対的検証 → 引用付き合成）。

## やってはいけないこと

- WebFetch の要約だけを根拠に「ソースにこう書いてある」と報告すること。
- 単一の二次情報（ブログ・まとめ）を一次情報として扱うこと。
- 確認していないことを確認済みのように断定すること。
