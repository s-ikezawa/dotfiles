# ユーザーストーリー管理コマンド

このコマンドはユーザーストーリーの作成とプロダクトマネージャー視点でのレビューを行います。
作成されたストーリーは `docs/requirements/user-stories/` ディレクトリに保存されます。

## コマンド実行

### /story

新しいユーザーストーリーを作成し、PMレビューを実施します。

**実行手順:**

1. 既存のストーリーIDを確認して、次のIDを自動採番：
   - `docs/requirements/user-stories/` ディレクトリ内のすべての `STORY-XXX.md` ファイルをスキャン
   - 最大のIDを見つけて +1 する
   - ファイルが存在しない場合は `STORY-001` から開始
   - 例: STORY-001, STORY-003, STORY-007 が存在する場合 → 次は STORY-008

2. 以下の情報を順番に収集してください：
   - **タイトル**: 機能の簡潔な名前
   - **As a（として）**: ユーザーロール
   - **I want（したい）**: 要求内容
   - **So that（なぜなら）**: 目的・価値
   - **詳細説明**: （オプション、Enterでスキップ可）

3. **PMレビュー実施** - 以下のプロンプトを使用してレビューを実施：

```
あなたは経験豊富なプロダクトマネージャーです。
以下のユーザーストーリーをレビューし、改善提案を行ってください。

## レビュー対象のストーリー
- ID: {ID}
- タイトル: {タイトル}
- As a（として）: {As a}
- I want（したい）: {I want}
- So that（なぜなら）: {So that}
- 詳細: {詳細説明}

## レビューの観点
1. **価値の明確性**: ユーザーにとっての価値が明確か
2. **スコープの適切性**: 1つのストーリーとして適切な粒度か
3. **測定可能性**: 成功を測定できる基準が定義可能か
4. **実現可能性**: 技術的・リソース的に実現可能か
5. **優先度の妥当性**: ビジネス価値に対して適切な優先度か

## 以下の形式でフィードバックを提供してください：

### 📊 総合評価
[5段階評価: ⭐⭐⭐⭐⭐]

### ✅ 良い点
- [良い点1]
- [良い点2]

### 🔧 改善提案
- [改善点1とその理由]
- [改善点2とその理由]

### 📝 推奨される受け入れ基準
- [ ] [具体的な基準1]
- [ ] [具体的な基準2]
- [ ] [具体的な基準3]

### 🎯 推奨優先度
[Must Have / Should Have / Could Have / Won't Have] - [理由]

### 💡 追加の考慮事項
- [技術的な考慮事項]
- [依存関係の確認事項]
```

4. PMレビュー結果を表示し、改善フィードバックループを開始：

**フィードバックループ処理:**

```
📋 PMレビューが完了しました

[レビュー結果を表示]

改善提案がある場合:
━━━━━━━━━━━━━━━━━━━━━━━━━━
🔧 以下の改善提案があります：
1. [改善提案1]
2. [改善提案2]

これらの改善を適用しますか？ (yes/no/skip): 
━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**yes を選択した場合:**
- 各改善提案について個別に対応
- 改善内容を入力してもらい、ストーリーを更新
- 更新後、再度PMレビューを実施
- 新たな指摘事項があれば繰り返し
- 指摘事項がなくなるまでループ

**no を選択した場合:**
- 特定の改善提案について理由を記録
- 次の改善提案へ進む

**skip を選択した場合:**
- 現在の内容でファイル保存へ進む

**改善ループの例:**
```
改善提案 1/2: 「ユーザーロールが曖昧です」
現在の値: 一般ユーザー
推奨: より具体的なペルソナ（例：初回購入を検討している20-30代の会社員）

修正しますか？ (yes/no): yes
新しい値を入力してください: 初回購入を検討している20-30代の会社員

✅ 更新されました

改善提案 2/2: 「成功基準が不明確です」
現在の値: キーワードで検索できる
推奨: 検索結果が2秒以内に表示され、関連度順にソートされる

修正しますか？ (yes/no): yes
新しい値を入力してください: 2秒以内に検索結果を関連度順で表示する

✅ 更新されました

━━━━━━━━━━━━━━━━━━━━━━━━━━
📊 再レビュー中...
━━━━━━━━━━━━━━━━━━━━━━━━━━

[再レビュー結果]

✨ 改善提案はありません。ストーリーは十分な品質です！
```

5. **最終確認とファイル保存:**

指摘事項がなくなった場合、または skip を選択した場合：

```
最終的なストーリーを確認してください：

ID: {ID}
タイトル: {タイトル}
As a（として）: {最終的なAs a}
I want（したい）: {最終的なI want}
So that（なぜなら）: {最終的なSo that}

これで保存してよろしいですか？ (yes/no): 
```

```markdown
# {ID}: {タイトル}

## ストーリー
**As a（として）** {As a}  
**I want（したい）** {I want}  
**So that（なぜなら）** {So that}

## 詳細
{詳細説明 または "（後で追加）"}

## 受け入れ基準
{PMレビューで推奨された基準、または}
- [ ] （後で追加）

## 技術メモ
{PMレビューの技術的考慮事項、または}
（後で追加）

## 依存関係
{PMレビューで指摘された依存関係、または}
（後で追加）

## 優先度
**{PMレビューで推奨された優先度、または「未設定」}**

---

## PMレビュー結果

### 最終評価
{最終評価}

### レビューサイクル
- 初回レビュー: {初回の指摘事項数}件の改善提案
- 改善実施: {改善した項目数}件
- 最終レビュー: {最終的な評価}

{改善した項目のリスト}
```

7. 完了メッセージを表示：
   ```
   ✅ ストーリー {自動採番されたID} を作成しました
   📁 保存先: docs/requirements/user-stories/{ID}.md
   🎯 {改善サイクル数}回の改善を経て品質が確保されました
   ```

## 長いコンテキスト対策

コンテキストが長くなる場合の対処法：

### オプション1: 段階的保存方式
1. 初回レビュー後に一度ファイルを保存
2. 保存したファイルを読み込んで改善を継続
3. 改善の都度、ファイルを更新

```
初回レビュー完了後:
━━━━━━━━━━━━━━━━━━━━━━━━━━
📁 一時保存しました: docs/requirements/user-stories/{ID}_draft.md
改善を続けますか？ (yes/no): yes

[ファイルを読み込んで改善継続]
━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### オプション2: 要約方式
- 改善履歴は要点のみを保持
- 詳細なレビュー内容は別ファイルに保存
- メインファイルには最終結果のみ記載

```
docs/requirements/user-stories/
├── STORY-001.md (最終版)
└── reviews/
    └── STORY-001_review_log.md (詳細な改善履歴)
```

## 自動採番の仕様

- `docs/requirements/user-stories/` ディレクトリ内のファイル名から最大のIDを検出
- パターン: `STORY-[0-9]{3}` の数値部分を抽出
- 最大値 + 1 を3桁ゼロパディングで新しいIDとする
- ディレクトリが空または存在しない場合は `STORY-001` から開始
- 途中のIDが欠番でも問題なく動作

## バリデーション

- 必須項目（タイトル、As a、I want、So that）が入力されていること
- 自動採番されたIDのファイルが存在しないことを確認
- PMレビューで指摘された問題がある場合は警告表示

## ディレクトリ構造

```
project/
├── .claude/
│   └── commands/
│       └── story.md (このファイル)
├── docs/
│   └── requirements/
│       └── user-stories/
│           ├── STORY-001.md
│           ├── STORY-002.md
│           └── ...
```

## エラーハンドリング

- ディレクトリが存在しない場合は自動作成
- ファイルの書き込み権限がない場合はエラーメッセージを表示
- 必須項目が未入力の場合は再入力を求める
- PMレビューでクリティカルな問題が発見された場合は警告を表示
